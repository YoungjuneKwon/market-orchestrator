# KIS 계좌 관리 시스템 작업 가이드

이 문서는 `/contents/workspace/kis` 경로에서 한국투자증권 계좌 관리 시스템을 구축할 때, 에이전트(또는 개발자)가 일관된 방식으로 작업하기 위한 기준 문서입니다.

## 1) 작업 목표

- 한국투자증권 Open API를 활용하여 계좌 조회, 잔고/주문 관리, 기본 자동화 작업이 가능한 로컬 시스템을 구축합니다.
- 초기 단계에서는 인증/환경 구성 안정화, 이후 기능 단위 모듈화(인증, 시세, 잔고, 주문, 로그) 순서로 확장합니다.

## 2) 레퍼런스 저장소

- 원본: `https://github.com/koreainvestment/open-trading-api`
- 로컬 클론 경로: `/contents/workspace/kis/reference/open-trading-api`
- 본 프로젝트는 위 저장소를 참고하여 구조와 인증 방식, API 호출 패턴을 재사용/응용합니다.

## 3) Open API 레퍼런스 분석 요약

### 3.1 저장소 구조 핵심

- `examples_llm/`: API 기능별 최소 호출 예제(LLM 친화 구조)
- `examples_user/`: 실사용 관점 통합 예제(카테고리별 함수/예제)
- `kis_auth.py`: 인증, 토큰 관리, 공통 호출/환경 전환 로직
- `strategy_builder/`, `backtester/`, `MCP/`: 전략 설계/백테스트/도구 연계 확장 영역

### 3.2 시스템 구성에 직접 유용한 포인트

- 인증 분리: 실전(`prod`) / 모의(`vps`) 환경을 코드 레벨에서 구분해 운용 가능
- 기능 분리: 상품군/기능군별 함수 파일 구조가 명확하여 모듈화에 유리
- 실행 패턴: 공통 인증 후 REST/WebSocket 기능을 호출하는 형태로 표준화되어 있음
- 환경 파일: 샘플은 `kis_devlp.yaml` 사용 중심이며, 본 프로젝트는 `.env` 기반으로 관리

### 3.3 기술 스택/요구사항 참고

- Python `>=3.11`
- 주요 의존성: `requests`, `websockets`, `pyyaml`, `pandas`, `pycryptodome` 등
- 샘플 저장소는 `uv` 기반 의존성 관리(`pyproject.toml`, `uv.lock`)를 사용

## 4) 민감정보(.env) 관리

민감정보는 다음 파일에 저장되어 있습니다.

- 파일 경로: `/contents/workspace/kis/.env`
- 확인된 환경변수 키:
  - `KIS_API_KEY`
  - `KIS_API_SECRET`
  - `KIS_CANO`
  - `KIS_ACNT_PRDT_CD`

보안 원칙:

- `KIS_API_KEY`, `KIS_API_SECRET` 값은 문서/로그/커밋에 평문으로 기록하지 않습니다.
- `.env`는 Git 커밋 대상에서 제외합니다(`.gitignore` 유지).
- 출력이 필요한 경우 키 값은 마스킹(`****`) 처리합니다.

## 5) 권장 구현 순서

1. `.env` 로딩 모듈 작성(환경변수 검증 포함)
2. 토큰 발급/갱신 모듈 작성(실패 재시도/만료 처리)
3. 계좌 기본 조회(잔고/예수금) API 래퍼 작성
4. 주문(현금 매수/매도) 래퍼 작성 및 모의투자 검증
5. 공통 로깅/예외 처리 및 실행 스크립트 정리

## 6) 작업 규칙

- 기본 개발 기준은 레퍼런스 저장소의 호출 패턴을 따르되, 본 프로젝트에서는 `.env` 중심 설정으로 단순화합니다.
- 신규 기능은 "단일 API 테스트 가능 파일" + "재사용 가능한 함수 모듈" 형태로 같이 유지합니다.
- 기능 추가 시 입력 파라미터, 응답 주요 필드, 오류코드를 문서화합니다.

### 6.1) 에이전트/CLI 동적 실행 시 토큰 재사용 규칙

- 자동화 스크립트 실행뿐 아니라, 에이전트 요청으로 CLI가 즉시 계좌 조회를 수행하는 경우에도 동일한 토큰 재사용 정책을 적용합니다.
- 접근 토큰은 최근 24시간 내 이미 발급된 값이 있으면 재발급하지 않고 재사용하는 것을 기본 정책으로 합니다.
- 토큰 발급 시 `access_token`, `access_token_issued_at`(UTC ISO-8601)를 로컬 임시 상태 파일(예: `/tmp/kis_access_token_state.json`)에 기록하고, 다음 실행 시 해당 파일을 우선 조회합니다.
- 임시 상태 파일의 발급 시각이 24시간 이내이면 파일의 토큰으로 작업을 처리하고, 24시간이 지난 경우에만 재발급을 시도합니다.
- 이 정책은 강제 청산/종목 추천/단건 계좌 조회 등 호출 주체와 무관하게 공통 적용 대상으로 간주합니다.
- 토큰 값은 로그/문서/커밋에 평문으로 남기지 않으며, 필요한 경우 마스킹 처리합니다.

## 7) 확장 목표(모노레포: Python + Node.js + Frontend)

- 새 저장소는 모노레포로 구성하며, Python/Node.js 스크립트와 웹/앱 UI를 한 저장소에서 관리합니다.
- 자동화 스크립트는 1분/5분 단위 스케줄 실행을 기본으로 하며, 핵심 작업은 다음을 포함합니다.
  - 계좌 잔고 조회
  - 시세 조회
  - 투자 안전 기능(강제 손절 매도 등)
  - 간단한 알고리즘 기반 종목 선정 및 매수 주문
- 프론트엔드는 상태 모니터링(잔고/포지션/신호/주문), 수동 개입(긴급 매도/중지), 실행 로그 확인이 가능해야 합니다.

## 8) 멀티 브로커 확장 전략(시장 인터페이스 추상화)

- 초기 브로커는 한국투자증권 Open API를 사용하되, 타 증권사 API 연동을 고려해 공통 인터페이스를 우선 설계합니다.
- 인터페이스 레이어는 최소 다음 도메인으로 분리합니다.
  - 인증(Auth)
  - 시세(Market Data)
  - 계좌(Account/Balance)
  - 주문(Order/Cancel)
  - 리스크/안전(Risk Guard)
- 전략/스케줄/프론트엔드는 브로커 구현체가 아닌 공통 인터페이스에 의존하도록 유지합니다.

## 9) 시크릿/배포 운영 원칙(GitHub Secrets 중심)

- 개인 민감정보 및 API 키는 코드/레포에 포함하지 않고, GitHub Secrets 또는 각 사용자 `.env`로만 주입합니다.
- 타 사용자가 저장소를 내려받아도 본인 키만 설정하면 즉시 실행 가능하도록 템플릿 기반 구성을 유지합니다.
  - 필수 환경변수 목록은 `.env.example` 또는 `docs/setup.md`에 문서화
  - 런타임 시작 시 필수 키 검증 및 누락 시 명확한 오류 메시지 제공
- CI/CD 및 자동 실행 워크플로우는 시크릿 미설정 시 안전하게 실패(fail-fast)하도록 구성합니다.

## 10) 권장 모노레포 예시 구조

- `apps/web`: 웹 대시보드(모니터링/제어 UI)
- `apps/mobile` 또는 `apps/desktop`: 앱 클라이언트(선택)
- `services/scheduler`: 1분/5분 작업 오케스트레이션
- `services/trading-bot`: 전략 실행, 주문, 리스크 가드
- `packages/broker-core`: 브로커 공통 인터페이스/도메인 모델
- `packages/broker-kis`: 한국투자증권 API 구현체
- `packages/shared`: 공통 타입, 설정 로더, 로깅, 유틸
